当测试工具越来越多，脚本越来越多，团队很快会遇到第二个问题：脚本能跑，但不好维护。

每个人都有自己的写法，环境配置散落在脚本里，接口请求重复封装，断言方式不统一，报告格式不一致。时间一长，自动化脚本就变成了一堆难以维护的代码。

这时候就需要自动化框架。

## 自动化框架解决什么问题

自动化框架不是为了让代码看起来高级，而是为了解决这些问题：

- 用例如何组织
- 环境如何切换
- 请求如何封装
- 鉴权如何处理
- 测试数据如何准备
- 断言如何统一
- 报告如何生成
- 失败如何定位
- 公共能力如何复用

如果没有框架，自动化脚本越多，维护成本越高。

## 框架设计的核心原则

## 1. 业务和底层实现分离

用例应该尽量表达业务意图，而不是堆满底层实现细节。

比如接口测试用例里，不应该到处写 token 获取、签名计算、请求头拼接、环境地址判断。

这些应该封装在客户端或公共方法里。

## 2. 配置和代码分离

环境地址、账号、数据库连接、开关配置，不应该写死在代码里。

更合理的方式是：

```text
config/
  dev.yml
  test.yml
  staging.yml
```

这样同一套用例可以在不同环境执行。

## 3. 数据和用例分离

对于大量参数组合，可以使用数据驱动。

测试逻辑保持稳定，测试数据单独维护。这样能减少重复代码，也方便扩展边界场景。

## 4. 断言要有业务含义

自动化用例不能只断言状态码是 200。

更重要的是断言业务结果：

- 任务是否创建成功
- 状态是否正确流转
- 数据库是否写入正确
- 回调是否触发
- 错误码是否符合预期
- 不该发生的副作用是否没有发生

断言越贴近业务，自动化价值越高。

## 5. 失败要容易定位

自动化失败后，要能快速判断：

- 是环境问题
- 是数据问题
- 是脚本问题
- 是产品缺陷
- 是第三方依赖问题

因此框架需要保留必要日志、请求响应、执行参数和错误上下文。

## 接口自动化框架的基础模块

一个接口自动化框架通常可以包括：

```text
api-tests/
  config/
  clients/
  testcases/
  data/
  utils/
  reports/
  conftest.py
```

各模块职责可以这样划分：

- config：环境配置
- clients：接口请求封装
- testcases：测试用例
- data：测试数据
- utils：公共工具
- reports：测试报告
- conftest.py：fixture、初始化、清理逻辑

这个结构不一定固定，但职责要清楚。

## UI 自动化框架的基础模块

UI 自动化可以采用 Page Object 或类似结构：

```text
ui-tests/
  pages/
  flows/
  testcases/
  data/
  utils/
  reports/
```

其中：

- pages 封装页面元素和基础操作
- flows 封装业务流程
- testcases 编写测试用例
- data 管理测试数据
- utils 提供通用工具

UI 自动化更容易受到页面变化影响，所以封装和定位策略尤其重要。

## 框架和平台的关系

测试平台不是替代自动化框架，而是管理和调度框架。

框架负责：

- 用例代码
- 执行逻辑
- 环境配置
- 数据准备
- 断言和报告

平台负责：

- 选择用例
- 创建任务
- 调度执行
- 展示报告
- 管理环境
- 分析趋势

如果没有稳定框架，平台就只是一个空壳。平台再漂亮，底层脚本不稳定，也没有价值。

## 框架阶段的交付标准

一个自动化框架是否成熟，可以看几个标准：

- 新人能否快速写出用例
- 用例是否有统一结构
- 环境切换是否简单
- 公共方法是否复用
- 报告是否清晰
- 失败是否容易定位
- 是否能接入 CI
- 是否能被平台调度

自动化框架的目标不是复杂，而是稳定、清晰、可维护。

## 结语

如果说测试工具化解决的是“重复操作”，那么自动化框架解决的是“规模化维护”。

没有框架，脚本越多越乱；有了框架，脚本才能变成可持续维护的测试资产。

测试平台的骨架，往往就是一套稳定的自动化框架。
