AI Agent 真正有价值的地方，不只是会回答问题。

更重要的是，它能连接工具、读取资料、调用系统、执行流程，并把经验沉淀成可复用能力。

在这个过程中，经常会遇到两个概念：

- MCP
- Skill

它们都和 AI Agent 的能力扩展有关，但解决的问题不一样。

简单说：

```text
MCP 解决“AI 如何连接外部工具和数据源”
Skill 解决“AI 如何稳定复用一套方法、流程和经验”
```

理解这两个概念，有助于我们把 AI 从聊天工具，逐步变成可以参与测试设计、自动化执行、接口验证、日志分析和知识沉淀的工程助手。

## 一、MCP 是什么

MCP 全称是 Model Context Protocol。

它是一个开放协议，用来标准化 AI 应用和外部系统之间的连接方式。

MCP 官方文档把它描述为：MCP 是一个开放协议，用于把应用程序中的上下文提供给大模型；它类似 AI 应用的 USB-C 接口。

这个比喻很直观。

过去不同 AI 工具要连接不同系统，往往需要分别开发适配逻辑。

比如连接数据库写一套。

连接文件系统写一套。

连接浏览器写一套。

连接 GitHub 写一套。

连接企业内部平台又写一套。

MCP 的目标，就是提供一种统一的连接方式，让 AI 应用可以通过标准协议访问外部能力。

## 二、MCP 能做什么

MCP 主要让 AI Agent 获得外部上下文和工具能力。

常见能力包括：

- 读取文件
- 查询数据库
- 调用 API
- 操作浏览器
- 访问代码仓库
- 查询业务系统
- 获取文档内容
- 调用自动化测试工具
- 读取日志和监控数据
- 执行受控命令

从测试开发角度看，MCP 可以让 AI 不再只依赖用户手动粘贴内容。

它可以直接获取更接近真实工作的上下文。

比如：

```text
读取需求文档
分析接口定义
查看自动化测试代码
查询测试报告
读取失败日志
调用测试执行平台
打开浏览器验证页面
查询数据库中的测试数据
```

这样 AI 才能从“根据描述猜测”，变成“基于实际上下文分析”。

## 三、为什么要使用 MCP

使用 MCP 的核心原因，是降低 AI 接入外部系统的成本。

如果没有统一协议，每个 AI 应用都要单独适配每个工具。

这会带来几个问题。

第一，接入成本高。

每连接一个系统，都要重新设计认证、数据读取、工具调用和结果返回。

第二，能力难复用。

一个工具接入了某个 AI 客户端，不代表另一个 AI 客户端也能用。

第三，边界不清楚。

AI 能访问什么、能调用什么、能返回什么，如果没有标准接口，就很难治理。

第四，安全和权限难控制。

外部系统连接越多，越需要明确哪些能力可以暴露给 AI，哪些操作需要限制。

MCP 的价值，就是把外部工具和数据源标准化暴露给 AI 应用。

对于测试工作来说，这非常重要。

因为测试不是孤立的。

测试需要连接：

- 需求
- 用例
- 接口
- 数据库
- 日志
- 缺陷系统
- 自动化平台
- 测试环境
- 监控系统

MCP 可以成为 AI Agent 接入这些系统的一种标准入口。

## 四、MCP 的基本组成

从使用者视角看，MCP 通常包括几个角色。

第一是 MCP Host。

也就是运行 AI Agent 的应用，比如编辑器、桌面客户端、聊天客户端或其他 AI 工具。

第二是 MCP Client。

它负责在 Host 内部和 MCP Server 建立连接。

第三是 MCP Server。

它负责把某个外部系统的能力暴露出来。

比如一个文件系统 MCP Server，可以提供读取文件、列目录等能力。

一个数据库 MCP Server，可以提供查询数据库的能力。

一个浏览器 MCP Server，可以提供打开页面、点击、截图等能力。

第四是外部系统。

也就是文件、数据库、API、浏览器、代码仓库、业务平台等真实资源。

可以简单理解为：

```text
AI Agent -> MCP Client -> MCP Server -> 外部系统
```

MCP Server 像一个适配器，把外部系统包装成 AI 可以理解和调用的能力。

## 五、MCP 的能力形态

MCP 中常见的能力形态包括：

- Tools
- Resources
- Prompts

Tools 更像“可以被调用的动作”。

比如查询数据库、读取日志、调用接口、打开网页、执行测试。

Resources 更像“可以读取的上下文资源”。

比如文件内容、文档、配置、数据表结构、测试报告。

Prompts 更像“可复用的提示模板”。

比如生成测试用例的提示词、分析缺陷的提示词、总结报告的提示词。

这三类能力让 AI Agent 不只是聊天，而是能围绕外部系统执行任务。

## 六、MCP 在测试中的使用场景

MCP 对测试开发尤其有价值。

因为测试工作天然需要大量上下文。

### 1. 需求分析

AI 可以通过 MCP 读取需求文档、接口文档和历史用例。

然后辅助生成：

- 测试点
- 风险清单
- 验收标准
- 边界场景
- 用例初稿

### 2. 接口测试

AI 可以读取 OpenAPI 文档、接口定义、请求示例和历史测试脚本。

然后生成接口测试建议：

- 参数校验
- 权限测试
- 状态流转
- 幂等测试
- 异常测试
- 回调测试

如果 MCP Server 接入了测试平台，AI 还可以进一步调用测试任务。

### 3. 自动化测试

AI 可以读取自动化测试代码。

然后分析：

- 用例结构是否合理
- 断言是否完整
- 测试数据是否稳定
- 失败原因可能是什么
- 哪些场景适合补自动化

### 4. 缺陷分析

AI 可以读取日志、截图、接口响应和测试报告。

然后输出：

- 缺陷摘要
- 复现路径
- 疑似原因
- 影响范围
- 回归建议

### 5. 测试报告

AI 可以从测试执行平台、缺陷系统和日志系统获取数据。

然后生成：

- 测试结论
- 风险说明
- 阻塞问题
- 遗留问题
- 上线建议

这类能力如果只靠人工复制粘贴，效率会很低。

MCP 的价值，就是让 AI 能直接接触这些上下文。

## 七、什么是 Skill

Skill 可以理解为给 AI Agent 使用的一套能力包。

它通常包含：

- 任务说明
- 操作流程
- 使用规则
- 示例
- 模板
- 脚本
- 参考资料
- 输出格式要求

如果说 MCP 是连接外部系统的接口，那么 Skill 更像是 AI 做某类工作的说明书和工具箱。

比如一个“接口测试设计 Skill”可以规定：

```text
先读取接口文档
再识别鉴权方式
再分析参数规则
再设计正向、反向、边界、权限、幂等、异常场景
最后输出 Markdown 测试清单
```

一个“测试报告生成 Skill”可以规定：

```text
先读取测试结果
再统计执行情况
再提取阻塞缺陷
再总结风险
最后输出测试结论和上线建议
```

Skill 的重点不是连接工具，而是让 AI 按稳定的方法做事。

## 八、Skill 能做什么

Skill 适合沉淀重复性工作流程。

常见用途包括：

- 需求分析工作流
- 测试用例设计工作流
- 接口测试检查清单
- 自动化脚本生成规范
- 缺陷分析模板
- 测试报告模板
- 代码评审规则
- 文档改写规范
- 数据分析流程
- 周报生成流程

测试团队里很多经验都可以沉淀成 Skill。

比如：

```text
如何评审测试用例
如何分析接口测试风险
如何判断自动化用例是否值得写
如何从缺陷复盘生成回归用例
如何把测试报告写成管理者能读懂的结论
```

这些经验如果只停留在个人脑子里，就很难复用。

写成 Skill 后，AI Agent 就可以按照统一流程执行。

## 九、Skill 与 MCP 的区别

MCP 和 Skill 经常一起出现，但它们不是一回事。

可以这样区分：

```text
MCP：连接外部系统
Skill：沉淀工作方法
```

更具体一点：

| 对比项 | MCP | Skill |
| --- | --- | --- |
| 关注点 | 工具和数据连接 | 方法和流程复用 |
| 解决问题 | AI 如何访问外部系统 | AI 如何稳定完成某类任务 |
| 典型内容 | Tools、Resources、Prompts | 指令、模板、脚本、参考资料 |
| 使用场景 | 查数据库、读文件、调 API、操作浏览器 | 生成用例、写报告、分析缺陷、总结文档 |
| 更像什么 | 标准接口 / 适配器 | 操作手册 / 能力包 |

一个例子更容易理解。

如果 AI 要生成接口测试用例：

MCP 负责让 AI 读取 OpenAPI 文档、访问接口定义、查询历史用例。

Skill 负责告诉 AI 应该按照什么方法设计接口测试，比如要覆盖参数、鉴权、幂等、状态流转、错误码和回调。

两者配合起来，AI 才能既有上下文，又有方法。

## 十、为什么要使用 Skill

使用 Skill 的核心原因，是让 AI 输出更稳定、更符合团队经验。

如果每次都临时写提示词，会有几个问题。

第一，输出不稳定。

同样的任务，不同人提问方式不同，AI 输出质量也不同。

第二，经验难复用。

资深测试人员知道怎么设计用例、怎么评审风险、怎么写报告，但这些经验如果不沉淀，新人和 AI 都很难稳定使用。

第三，流程不可控。

AI 可能直接给结果，但跳过了关键步骤，比如没有先确认范围、没有识别风险、没有说明假设。

第四，团队标准不统一。

不同人用 AI 生成的内容格式不同、深度不同、质量标准不同，后续难维护。

Skill 可以把团队经验结构化。

它让 AI 知道：

- 先做什么
- 后做什么
- 必须检查什么
- 输出什么格式
- 哪些内容不能写
- 哪些风险必须说明
- 什么时候需要人工确认

这对测试团队非常重要。

因为测试工作不仅要快，还要可追踪、可复用、可审核。

## 十一、MCP 和 Skill 如何配合

比较理想的 AI 测试工作流是：

```text
MCP 提供上下文和工具
Skill 提供方法和流程
AI Agent 负责理解、规划、执行和总结
测试人员负责目标、边界和最终判断
```

比如一个 AI 辅助接口测试流程：

```text
通过 MCP 读取接口文档
通过 MCP 查询历史缺陷和历史用例
使用接口测试设计 Skill 生成测试点
使用自动化脚本 Skill 生成测试脚本
通过 MCP 调用测试执行平台
通过 MCP 读取执行结果和日志
使用测试报告 Skill 生成测试结论
测试人员审核风险和上线建议
```

这里 MCP 和 Skill 各自承担不同角色。

MCP 让 AI 能接触真实系统。

Skill 让 AI 按正确的方法工作。

## 十二、什么时候用 MCP，什么时候用 Skill

如果问题是“AI 需要访问某个外部系统”，优先考虑 MCP。

比如：

- 读取文件
- 查询数据库
- 调用接口
- 访问 Git 仓库
- 操作浏览器
- 获取日志
- 查询测试报告

如果问题是“AI 需要稳定完成某类任务”，优先考虑 Skill。

比如：

- 生成测试用例
- 总结测试报告
- 分析缺陷原因
- 改写技术文章
- 评审测试用例
- 设计测试策略
- 输出固定格式文档

如果一个任务既需要访问系统，又需要按固定方法执行，就应该 MCP 和 Skill 一起用。

## 十三、对测试工程师的意义

MCP 和 Skill 的出现，对测试工程师很有价值。

它们让 AI 测试从“聊天式辅助”走向“工程化协作”。

过去 AI 更多是：

```text
我粘贴需求，你帮我生成测试点。
```

未来更可能是：

```text
AI 自动读取需求、接口、历史用例和缺陷记录。
按照团队测试设计 Skill 生成测试策略和用例。
调用测试平台执行部分验证。
读取日志和报告。
生成风险结论。
测试人员审核并决策。
```

这会改变测试工程师的工作重点。

测试人员不只是写提示词，而是要设计：

- 哪些系统可以接入 AI
- 哪些数据可以提供给 AI
- 哪些流程适合 Skill 化
- 哪些判断必须人工审核
- 哪些测试资产可以长期沉淀
- 哪些能力可以接入测试平台

这也是 AI 测试工程化的关键。

## 十四、最后总结

MCP 是协议。

它解决 AI 如何连接外部系统、工具和数据源。

Skill 是能力包。

它解决 AI 如何复用团队经验、流程和方法。

MCP 让 AI 有手和眼睛。

Skill 让 AI 有做事的方法。

两者结合，AI Agent 才能真正参与复杂的软件测试工作。

对于测试开发来说，未来值得建设的不只是自动化脚本，还有：

- 可接入的测试工具链
- 可复用的测试知识库
- 可执行的测试工作流
- 可治理的 AI Skill
- 可观测的测试结果和风险闭环

这才是 AI 测试从“能帮忙”走向“能协作”的关键。
