# 测试用例设计 Skill 设计文档

## 一、设计背景

需求测试分析解决“需求是否清楚、功能如何拆、风险在哪里”。

测试策略设计解决“测什么、测多深、如何取舍”。

测试用例设计则要进一步解决“具体怎么验证”。

这个 Skill 的目标不是简单把测试点扩写成步骤，而是把测试策略、功能树、测试点、质量特性、业务规则和验收标准转化为可执行、可判定、可维护、可追溯的测试用例。

测试用例应该能服务后续执行、评审、回归、自动化和质量资产沉淀。

## 二、Skill 定位

建议名称：

```text
测试用例设计 Skill
```

建议英文名：

```text
test-case-design
```

核心定位：

```text
基于需求测试分析报告和测试策略设计报告，结合功能树、测试点、质量特性、验收标准、业务规则、接口文档、UI 原型和历史缺陷，生成分层、可执行、可判定、可追溯的测试用例集。
```

该 Skill 不负责：

- 重新评审需求完整性
- 制定测试策略
- 直接生成自动化脚本
- 替代测试用例评审

如果上游需求不可测试，或存在 P0 待确认问题，该 Skill 应优先提示无法生成完整用例。

## 三、适用场景

适合使用该 Skill 的场景包括：

- 新需求测试用例设计
- 迭代需求用例设计
- 接口测试用例设计
- 回归测试用例补充
- 冒烟用例集设计
- 权限和状态流转用例设计
- AI 辅助从测试点生成用例
- 测试资产库初始化

不建议直接用于：

- 需求不清晰时强行生成完整用例
- 自动化脚本生成
- 性能压测脚本设计
- 安全专项深度测试方案
- 上线准入判断

## 四、输入信息

### 1. 核心输入

建议至少提供：

```text
需求测试分析报告
测试策略设计报告
```

需求测试分析报告应包含：

- 需求摘要
- 可测试性判断
- 待确认信息
- 功能树
- 质量特性选择
- 测试点分析
- 风险提示
- 影响范围和回归建议

测试策略设计报告应包含：

- 测试目标
- 测试范围
- 测试深度
- 测试类型
- 回归范围
- 测试优先级
- 自动化候选
- 资源约束

### 2. 推荐输入

- PRD / 用户故事 / 变更说明
- UI 原型 / 交互说明
- 开发设计文档
- 接口文档
- 数据模型和状态说明
- 角色权限说明
- 验收标准
- 历史缺陷
- 历史用例
- 自动化用例集
- 测试数据模板

### 3. 输入缺失处理规则

如果缺少核心输入，Skill 应输出：

```text
无法生成完整测试用例的原因
缺失资料
对用例设计的影响
可以先生成的有限用例范围
```

如果缺少 UI 原型，可以继续生成业务逻辑用例，但应把页面交互细节列为待确认。

如果缺少接口文档，可以继续生成业务场景用例，但不应生成具体接口字段级用例。

如果缺少验收标准，应避免生成不可判定的预期结果。

## 五、输出目标

Skill 输出一份结构化测试用例设计报告。

报告需要包含：

- 用例设计摘要
- 用例分层说明
- 测试用例清单
- 冒烟用例
- 主流程用例
- 异常用例
- 边界用例
- 权限用例
- 状态流转用例
- 数据一致性用例
- 兼容性或易用性用例
- 历史缺陷回归用例
- 自动化候选用例
- 待确认信息
- 用例覆盖说明

## 六、设计流程

建议 Skill 按以下流程执行。

```text
读取需求测试分析报告
  ↓
读取测试策略设计报告
  ↓
确认可测试性和待确认信息
  ↓
确认测试范围、测试深度和测试优先级
  ↓
按功能树拆分用例结构
  ↓
按质量特性和测试类型补充场景
  ↓
使用测试设计方法生成具体用例
  ↓
补充前置条件、测试数据、步骤和预期结果
  ↓
标注优先级、用例类型、来源、回归建议和自动化候选
  ↓
检查用例可执行性、可判定性和重复性
  ↓
输出测试用例设计报告
```

## 七、用例分层规则

测试用例不应该平铺。

建议至少按以下层次组织：

### 1. 冒烟用例

用于快速判断版本是否具备继续测试条件。

特点：

- 数量少
- 优先级高
- 覆盖核心入口和主流程
- 预期结果明确
- 适合自动化

### 2. 主流程用例

用于验证需求的核心业务路径。

特点：

- 覆盖用户主要操作路径
- 覆盖核心业务规则
- 覆盖核心数据和状态变化

### 3. 异常用例

用于验证失败、错误、取消、重试、恢复等场景。

常见异常包括：

- 输入错误
- 权限不足
- 状态不允许
- 接口失败
- 网络异常
- 重复提交
- 外部依赖失败

### 4. 边界用例

用于验证临界值和极端情况。

常见边界包括：

- 最小值
- 最大值
- 刚好等于限制
- 超过限制
- 空值
- 长文本
- 特殊字符
- 批量上限

### 5. 权限用例

用于验证不同角色和权限边界。

需要覆盖：

- 有权限用户
- 无权限用户
- 不同角色差异
- 数据权限
- 接口越权
- 前后端权限一致性

### 6. 状态流转用例

用于验证对象状态变化。

需要覆盖：

- 初始状态
- 中间状态
- 结束状态
- 非法状态转换
- 取消、失败、重试、恢复
- 并发或重复操作后的状态一致性

### 7. 数据一致性用例

用于验证数据写入、更新、同步、展示和统计。

需要覆盖：

- 页面展示和接口返回一致
- 数据库状态正确
- 统计口径正确
- 缓存和异步同步一致
- 导入导出一致

### 8. 历史缺陷回归用例

用于防止历史问题复现。

来源包括：

- 历史缺陷
- 线上问题
- 缺陷复盘
- 高风险检查清单

## 八、测试设计方法

Skill 应根据需求类型选择合适的测试设计方法。

### 1. 等价类划分

适用于输入校验、参数组合、字段规则。

输出时应区分有效等价类和无效等价类。

### 2. 边界值分析

适用于长度、数量、金额、时间、次数、范围限制。

必须关注临界点附近的值。

### 3. 判定表

适用于多个条件组合影响结果的业务规则。

比如角色、状态、配置、开关、权限共同决定可操作项。

### 4. 状态迁移

适用于订单、任务、审批、账号、设备等有明确状态的对象。

需要覆盖合法流转和非法流转。

### 5. 场景法

适用于端到端业务流程。

应覆盖用户从开始到完成目标的完整路径。

### 6. 错误推测

适用于结合历史缺陷、复杂模块、经验风险补充用例。

### 7. 正交和组合

适用于多参数组合，但不能穷尽所有组合时。

需要结合风险选择代表性组合。

## 九、质量特性映射

用例设计应承接需求测试分析中的质量特性选择。

### 功能性

生成规则：

- 覆盖业务规则
- 覆盖输入输出
- 覆盖权限和状态
- 覆盖主流程和异常流程

### 可靠性

生成规则：

- 覆盖失败恢复
- 覆盖重试和幂等
- 覆盖异常中断
- 覆盖状态一致性

### 易用性

生成规则：

- 覆盖入口清晰度
- 覆盖提示文案
- 覆盖错误反馈
- 覆盖操作路径是否顺畅

### 效率

生成规则：

- 覆盖响应时间关注点
- 覆盖批量处理
- 覆盖大数据量
- 覆盖可能的性能退化场景

### 维护性

生成规则：

- 覆盖配置变更
- 覆盖日志和错误信息
- 覆盖规则扩展
- 覆盖问题定位所需信息

### 可移植性

生成规则：

- 覆盖多端
- 覆盖浏览器
- 覆盖系统版本
- 覆盖设备差异

不应默认六大特性都生成用例，必须根据策略选择重点。

## 十、用例字段规范

每条用例建议包含以下字段：

```text
用例 ID
所属模块
功能节点
用例标题
用例类型
优先级
前置条件
测试数据
操作步骤
预期结果
来源
关联质量特性
是否回归
是否冒烟
是否自动化候选
备注
```

字段说明：

- 来源：应能追溯到需求、功能树、测试点、历史缺陷或策略报告。
- 优先级：建议使用 P0 / P1 / P2。
- 用例类型：主流程、异常、边界、权限、状态、数据、回归、兼容等。
- 自动化候选：只标记建议，不直接生成脚本。

## 十一、优先级规则

### P0

必须执行。

包括：

- 核心业务流程
- 核心数据写入
- 权限安全边界
- 关键状态流转
- 冒烟用例
- 上线准入强相关场景

### P1

应执行。

包括：

- 主要异常流程
- 常见边界
- 常见用户路径
- 重要回归范围
- 历史缺陷相关场景

### P2

视资源执行。

包括：

- 低频场景
- 展示细节
- 辅助流程
- 低风险兼容场景

## 十二、自动化候选判断

适合标记为自动化候选：

- 高频回归
- 核心冒烟
- 预期结果明确
- 数据可构造
- 执行路径稳定
- 接口层可验证
- 历史缺陷回归

不适合立即自动化：

- 需求频繁变化
- UI 尚不稳定
- 需要强人工判断
- 测试数据难稳定
- 执行频率低且维护成本高

## 十三、输出模式

### 简版

适合快速生成核心用例。

输出：

- 冒烟用例
- P0 主流程用例
- 主要异常用例

### 标准版

适合常规需求。

输出：

- 完整分层用例
- 优先级
- 测试数据
- 预期结果
- 来源追溯
- 自动化候选

### 深度版

适合复杂或高风险需求。

输出：

- 标准版全部内容
- 状态迁移用例
- 权限矩阵用例
- 数据一致性用例
- 历史缺陷回归
- 组合场景
- 覆盖说明

## 十五、约束规则

- 需求不可测试时，不强行生成完整用例。
- 不把测试点机械扩写成重复用例。
- 不生成不可判定的预期结果。
- 不忽略异常、边界、权限、状态和数据一致性场景。
- 不默认所有质量特性都需要设计用例。
- 不把自动化候选等同于自动化脚本。
- 涉及内部信息时必须脱敏。

## 十六、输出质量验收标准

| 验收项 | 合格标准 |
| --- | --- |
| 可执行性 | 用例具备前置条件、步骤、数据和预期 |
| 可判定性 | 预期结果明确，能判断通过或失败 |
| 覆盖性 | 覆盖主流程、异常、边界、权限、状态和回归 |
| 分层合理 | 冒烟、主流程、异常、回归等层次清楚 |
| 优先级合理 | P0/P1/P2 与风险和测试策略一致 |
| 可追溯 | 用例能追溯到需求、测试点、策略或历史缺陷 |
| 可维护 | 用例不重复、不冗余，便于后续回归和自动化 |

## 十七、与其他 Skill 的关系

```text
需求测试分析 Skill
  ↓
测试策略设计 Skill
  ↓
测试用例设计 Skill
  ↓
测试用例评审 Skill
  ↓
测试数据设计 Skill
  ↓
测试执行辅助 Skill
```

测试用例设计 Skill 的输出，是用例评审、测试数据设计、自动化测试设计和测试执行辅助的核心输入。
