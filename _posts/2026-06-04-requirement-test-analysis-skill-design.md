# 需求测试分析 Skill 设计文档

## 一、设计背景

在实际测试工作中，需求往往不是天然完整、清晰、可测试的。

如果测试人员在需求不清楚的情况下直接设计用例，很容易出现几个问题：

- 用例看起来很多，但没有覆盖真实风险
- 需求歧义没有提前暴露，后续提测后才反复确认
- 主流程覆盖了，但异常流程、状态闭环和数据边界遗漏
- 测试点和需求之间缺少结构化关系
- AI 直接生成用例时，容易把不确定内容“自动补全”，造成误导

因此，这个 Skill 的定位不是直接生成测试用例，而是在需求阶段进行测试视角分析。

它的目标是让 AI 更早参与需求评审、功能拆解、风险识别和测试点设计，为后续用例设计、自动化设计和测试策略制定提供输入。

## 二、Skill 定位

建议名称：

```text
需求测试分析 Skill
```

建议英文名：

```text
requirement-test-analysis
```

核心定位：

```text
对需求进行测试视角分析，识别需求缺失、歧义和未闭环点，梳理业务模型、功能结构、数据关系和影响范围，并基于必要的软件质量特性生成测试点、风险提示和待确认问题。
```

这个 Skill 关注的是需求分析和测试设计前置工作，而不是完整测试用例生成。

它需要覆盖的不只是功能拆解，还包括业务理解、风险识别、测试范围控制、回归影响分析、优先级判断和后续测试策略输入。

## 三、适用场景

适合使用该 Skill 的场景包括：

- 新需求测试分析
- 需求评审前准备
- 产品需求文档测试视角审阅
- 接口需求测试分析
- 迭代需求变更影响分析
- 旧功能回归范围分析
- 测试范围和优先级初步判断
- AI 辅助测试点生成
- 用例设计前的功能结构拆解
- 需求不完整时的待确认问题整理
- 测试策略制定前的输入准备

不建议直接用于：

- 完整测试用例生成
- 自动化脚本生成
- 性能压测方案生成
- 安全专项测试方案生成
- 上线测试报告生成

这些可以作为后续 Skill 或下游流程。

## 四、输入信息

Skill 的输入不应该只限制为需求文档。

真实项目中的测试分析，往往需要综合产品、研发、设计、接口、数据、历史缺陷和项目约束等多类资料。

但实际工作中，这些资料可能不完整，也可能只有其中一部分。

因此输入文档需要分层设计：

```text
核心输入：没有它就无法进行基本分析
推荐输入：有它可以显著提升分析质量
可选输入：根据项目类型补充
缺失输入：记录为待确认信息，不强行脑补
```

### 1. 核心输入文档

核心输入是测试分析的最低基础。

至少应提供其中一种：

```text
需求描述 / PRD / 用户故事 / 变更说明
```

这些内容用于回答：

- 要做什么
- 为什么做
- 给谁用
- 涉及哪些功能
- 预期结果是什么
- 本次变更范围是什么

如果核心输入缺失，Skill 应停止生成详细测试点，只输出“无法完成有效测试分析”的原因和需要补充的信息。

### 2. 产品需求类文档

这类文档用于理解业务目标和功能规则。

```text
产品需求文档 PRD
用户故事 User Story
需求变更说明
业务规则说明
验收标准
范围说明
不在本次范围内的说明
业务流程说明
角色权限说明
运营规则说明
配置规则说明
```

重点关注：

- 业务目标
- 用户角色
- 主流程
- 异常流程
- 业务规则
- 验收标准
- 范围边界
- 新旧功能影响

### 3. 开发设计类文档

这类文档用于理解实现方式、系统影响范围和技术风险。

```text
开发设计文档
技术方案
系统架构图
模块设计说明
时序图
状态机设计
数据流设计
接口调用链路
任务调度设计
消息队列设计
缓存设计
异常处理设计
容错和重试设计
日志和监控设计
灰度和回滚方案
```

重点关注：

- 哪些模块被影响
- 调用链路是否清楚
- 状态变化是否闭环
- 异常恢复是否设计
- 是否存在异步、重试、补偿逻辑
- 是否有日志、监控和告警
- 是否影响性能、稳定性或维护性

开发设计文档对于可靠性、效率、维护性测试分析尤其重要。

如果只有 PRD，没有开发设计文档，Skill 应在报告中说明技术风险判断可能不足。

### 4. UI 和交互类文档

这类文档用于理解用户操作路径和页面体验。

```text
UI 原型
交互原型
页面设计稿
页面流程图
字段说明
组件状态说明
错误提示文案
空状态设计
加载状态设计
禁用状态设计
弹窗和二次确认说明
多端适配设计
```

重点关注：

- 用户入口是否明确
- 页面流程是否闭环
- 输入控件和字段规则是否清楚
- 错误提示是否明确
- 空状态、加载状态、禁用状态是否覆盖
- 操作成功和失败后反馈是否明确
- 移动端、Web 端或不同分辨率是否需要适配

UI 原型对于易用性、功能性和可移植性测试分析非常重要。

如果 UI 原型缺失，Skill 可以继续分析业务逻辑，但应把交互细节列为待确认信息。

### 5. 接口和协议类文档

这类文档用于分析接口行为、参数规则、鉴权、状态流转和系统间交互。

```text
OpenAPI 文档
接口定义文档
接口请求和响应示例
错误码文档
鉴权说明
签名规则
回调通知说明
Webhook 文档
消息协议文档
MQTT / WebSocket / RPC / gRPC 协议说明
第三方接口说明
接口变更记录
接口兼容性说明
```

重点关注：

- 请求参数是否完整
- 返回字段是否明确
- 必填和非必填规则是否清楚
- 错误码是否完整
- 鉴权和权限是否明确
- 重复请求是否幂等
- 超时、重试、回调是否说明
- 版本兼容是否说明
- 第三方依赖失败如何处理

接口文档对于接口测试、自动化测试和数据一致性分析非常关键。

### 6. 数据和模型类文档

这类文档用于理解数据结构、数据流转和一致性风险。

```text
数据字典
数据库表结构
字段说明
状态枚举说明
数据流图
数据同步规则
数据初始化规则
数据清理规则
统计口径说明
报表口径说明
缓存数据规则
数据权限说明
数据脱敏规则
```

重点关注：

- 数据从哪里来
- 数据写到哪里
- 状态如何变化
- 字段含义是否明确
- 是否存在统计口径
- 是否存在同步延迟
- 是否涉及缓存一致性
- 是否涉及数据权限和脱敏

如果数据类文档缺失，Skill 应降低对数据一致性结论的确定性，并输出待确认问题。

### 7. 质量约束类文档

这类文档用于判断是否需要分析性能、安全、稳定性、兼容性等非功能风险。

```text
性能指标
容量要求
并发要求
稳定性要求
可用性要求
安全要求
权限要求
合规要求
兼容性要求
浏览器和设备支持范围
操作系统支持范围
网络环境要求
SLA / SLO
监控和告警要求
```

重点关注：

- 是否有响应时间要求
- 是否有并发或容量要求
- 是否有稳定性要求
- 是否有安全和权限要求
- 是否有兼容性范围
- 是否有线上监控和告警要求

这些输入会影响六大质量特性的选择。

如果质量约束没有提供，Skill 不应默认所有非功能测试都需要深测，而应根据需求内容判断，并把缺失指标列为待确认信息。

### 8. 测试资产类文档

这类文档用于复用历史经验，避免重复遗漏。

```text
历史测试用例
历史测试报告
历史缺陷
缺陷复盘记录
线上问题记录
冒烟检查清单
回归用例集
自动化用例
接口测试集合
风险清单
测试数据模板
测试环境说明
```

重点关注：

- 历史上哪里容易出问题
- 哪些场景需要回归
- 哪些缺陷可能再次出现
- 哪些用例可以复用
- 哪些自动化脚本可能受影响
- 哪些测试数据可以复用

历史资产可以显著提升测试点的针对性。

没有历史资产时，Skill 应说明当前分析主要基于需求本身和通用测试经验。

### 9. 项目过程类信息

这类信息用于判断测试策略和交付风险。

```text
版本计划
提测时间
上线时间
发布范围
灰度计划
回滚方案
依赖系统排期
研发分支说明
测试环境可用性
项目风险记录
需求评审记录
会议纪要
变更记录
```

重点关注：

- 测试时间是否充足
- 是否存在压缩测试风险
- 依赖系统是否按时就绪
- 是否有灰度和回滚
- 需求是否频繁变更
- 测试环境是否稳定

这些信息不一定影响功能测试点，但会影响测试策略、风险提示和后续建议。

### 10. 输入文档优先级建议

不同场景下，输入文档优先级不同。

通用需求分析建议优先级：

```text
P0：需求文档 / 用户故事 / 变更说明
P0：验收标准
P1：UI 原型
P1：开发设计文档
P1：接口文档
P1：角色权限说明
P1：状态和数据规则
P2：历史用例和历史缺陷
P2：性能、安全、兼容性等质量约束
P2：测试环境和发布计划
```

接口类需求建议优先级：

```text
P0：接口定义文档
P0：业务规则和验收标准
P1：鉴权说明
P1：错误码文档
P1：状态流转说明
P1：回调或异步机制说明
P2：历史接口用例和缺陷
```

前端交互类需求建议优先级：

```text
P0：需求描述
P0：UI / 交互原型
P1：字段规则
P1：错误提示和异常状态
P1：接口文档
P2：兼容性要求
P2：历史体验问题
```

复杂系统类需求建议优先级：

```text
P0：需求文档
P0：开发设计文档
P1：架构图和时序图
P1：状态机和数据流
P1：接口和协议文档
P1：异常恢复和补偿设计
P2：日志监控和回滚方案
```

### 11. 输入缺失处理规则

输入信息不足时，Skill 必须遵循以下规则：

- 不强行脑补未提供的业务规则
- 不把推测内容当作事实
- 可以基于经验提出假设，但必须标记为“假设”
- 缺失信息要进入待确认信息
- 对测试分析影响大的缺失信息要标记为 P0 或 P1
- 如果缺失核心输入，只输出评审问题，不生成完整测试点
- 如果缺失设计文档，可以继续分析功能性，但要降低可靠性、效率、维护性结论的确定性
- 如果缺失 UI 原型，可以继续分析业务逻辑，但要把交互、易用性、可移植性相关问题列为待确认

### 12. 输入文档汇总表

| 文档类型 | 是否必需 | 主要用途 | 缺失影响 |
| --- | --- | --- | --- |
| 需求文档 / 用户故事 / 变更说明 | 必需 | 理解需求目标和功能范围 | 无法进行有效分析 |
| 验收标准 | 推荐 | 判断预期结果是否可判定 | 影响可测试性判断 |
| 开发设计文档 | 推荐 | 分析技术实现和系统影响 | 影响可靠性、效率、维护性分析 |
| UI 原型 / 交互设计 | 推荐 | 分析页面流程和易用性 | 影响交互和易用性测试点 |
| 接口文档 | 推荐 | 分析接口行为和数据交互 | 影响接口、权限、异常和幂等测试 |
| 数据模型 / 状态说明 | 推荐 | 分析数据一致性和状态闭环 | 影响状态流转和数据验证 |
| 质量约束文档 | 可选 | 分析性能、安全、兼容性等非功能需求 | 影响非功能测试深度 |
| 历史用例 / 历史缺陷 | 可选 | 复用经验和补充风险 | 影响测试点针对性 |
| 测试环境和发布计划 | 可选 | 判断交付和执行风险 | 影响策略和风险提示 |

## 五、输出目标

Skill 输出一份结构化的需求测试分析报告。

报告需要帮助测试人员和研发团队回答以下问题：

- 这个需求要解决什么问题
- 当前需求是否清楚
- 哪些信息缺失或有歧义
- 需求是否形成完整闭环
- 功能可以拆成哪些模块和子功能
- 本需求涉及哪些软件质量特性
- 哪些质量特性不需要重点测试
- 应该设计哪些测试点
- 哪些风险需要提前确认
- 当前是否具备进入用例设计的条件

## 六、分析流程

建议 Skill 按以下流程执行。

```text
读取需求
  ↓
提炼需求摘要
  ↓
进行需求评审
  ↓
输出待确认信息
  ↓
梳理业务模型和功能树
  ↓
分析数据、接口、状态和角色权限
  ↓
分析变更影响范围和回归范围
  ↓
选择需要分析的软件质量特性
  ↓
按功能树和质量特性生成测试点
  ↓
识别风险、优先级和自动化候选
  ↓
判断是否具备用例设计条件
  ↓
输出测试分析报告
```

## 七、需求评审维度

需求评审重点关注需求是否清晰、完整、闭环和可测试。

建议检查以下维度。

### 1. 业务目标

检查点：

- 需求要解决的问题是否明确
- 目标用户或使用角色是否明确
- 需求成功的判断标准是否明确
- 业务价值是否可以理解
- 核心业务对象是否明确
- 业务成功和失败的判定是否明确
- 对上下游业务流程的影响是否明确

### 2. 范围边界

检查点：

- 本次需求包含哪些功能
- 明确不包含哪些功能
- 是否影响旧功能
- 是否涉及多端、多角色、多系统
- 是否有灰度或阶段性交付范围
- 哪些内容需要重点测试
- 哪些内容可以轻量验证
- 哪些内容明确不在本次测试范围
- 哪些旧功能需要回归

### 3. 流程闭环

检查点：

- 主流程是否完整
- 异常流程是否说明
- 取消、失败、重试、恢复等场景是否说明
- 需求结束状态是否明确
- 用户操作后系统状态是否明确

### 4. 数据闭环

检查点：

- 输入数据来源是否明确
- 输出数据去向是否明确
- 数据状态变化是否明确
- 数据是否需要持久化
- 是否涉及数据同步、统计或展示
- 数据口径是否明确
- 数据权限是否明确
- 数据一致性要求是否明确
- 缓存、延迟、同步失败等场景是否说明

### 5. 状态闭环

检查点：

- 是否存在状态流转
- 初始状态是否明确
- 中间状态是否明确
- 结束状态是否明确
- 非法状态转换是否说明
- 状态异常后如何恢复
- 重复操作、撤销、取消、回滚后的状态是否明确
- 并发或异步场景下状态是否可能不一致

### 6. 权限和角色

检查点：

- 涉及哪些用户角色
- 不同角色可执行的操作是否明确
- 无权限访问如何处理
- 是否存在越权风险
- 前端权限和接口权限是否一致
- 数据权限和操作权限是否分开说明

### 7. 交互和提示

检查点：

- 用户操作入口是否明确
- 页面提示是否明确
- 异常提示是否明确
- 操作成功后反馈是否明确
- 是否需要二次确认

### 8. 依赖和约束

检查点：

- 是否依赖外部系统
- 是否依赖配置、开关或权限
- 是否依赖特定环境
- 是否涉及接口、任务、消息、回调等异步机制
- 外部依赖失败时如何处理
- 依赖系统版本或兼容要求是否明确
- 是否需要监控、日志或告警支持

### 9. 变更影响和回归范围

检查点：

- 本次变更影响哪些旧功能
- 是否影响已有接口、数据结构或配置
- 是否影响已有自动化用例
- 是否影响历史缺陷相关场景
- 是否影响核心冒烟链路
- 是否需要补充回归测试
- 是否需要调整测试数据或测试环境

### 10. 验收标准

检查点：

- 验收标准是否明确
- 预期结果是否可判定
- 是否存在模糊描述
- 是否能据此设计测试用例

## 八、待确认信息分级

待确认信息建议按优先级分级。

### P0：阻塞测试设计

这类信息不确认，就无法继续设计有效测试点。

例如：

- 核心业务规则缺失
- 主流程不完整
- 状态定义不明确
- 验收标准不可判定

### P1：影响测试范围和风险判断

这类信息不确认，会影响测试深度和覆盖范围。

例如：

- 是否影响旧功能
- 是否涉及权限差异
- 异常流程如何处理
- 是否需要兼容多端或多环境

### P2：影响测试细节

这类信息不确认，不一定阻塞整体分析，但会影响具体用例设计。

例如：

- 文案提示
- 排序规则
- 默认值
- 页面展示细节

## 九、功能树设计

功能树用于把自然语言需求拆解成可测试结构。

但复杂需求只用功能树可能不够。

如果需求涉及状态、数据、权限、接口或异步任务，Skill 应在功能树之外补充必要的业务模型。

建议功能树至少包含：

```text
一级功能
- 二级功能
  - 具体能力
  - 操作入口
  - 输入条件
  - 输出结果
  - 状态变化
  - 异常处理
```

如果需求较简单，可以输出轻量功能树。

如果需求复杂，可以按模块、角色、流程或状态进行拆分。

复杂需求建议补充以下模型：

```text
业务流程：用户或系统完成目标的完整路径
状态模型：对象从初始状态到结束状态的流转关系
角色权限模型：不同角色可以查看或操作的范围
数据流模型：数据来源、写入、同步、展示和清理路径
接口交互模型：前端、后端、第三方或异步系统之间的调用关系
影响范围模型：本次变更影响的新旧功能、接口、数据和自动化资产
```

这些模型不一定每次都输出。

如果需求简单，功能树即可。

如果需求复杂，Skill 应选择必要模型补充分析，并说明选择原因。

示例结构：

```text
账号登录
- 密码登录
  - 输入账号和密码
  - 校验账号状态
  - 校验密码正确性
  - 登录成功后进入首页
  - 登录失败时提示错误原因
- 登录状态保持
  - 刷新页面保持登录
  - Token 过期后重新登录
- 退出登录
  - 清理登录状态
  - 返回登录页
```

## 十、软件质量特性选择

该 Skill 使用软件质量六大特性作为测试点分析框架。

六大特性包括：

```text
功能性
可靠性
易用性
效率
维护性
可移植性
```

但不能机械套用六大特性。

Skill 必须根据需求内容选择需要重点分析的质量特性。

每个质量特性需要输出：

```text
是否需要分析：是 / 否
选择原因：
测试关注点：
```

### 1. 功能性

适用情况：

- 需求涉及新增功能
- 需求涉及业务规则变化
- 需求涉及接口、数据、权限或状态变化

关注点：

- 功能是否实现
- 规则是否正确
- 输入输出是否正确
- 权限是否符合预期
- 状态流转是否正确

### 2. 可靠性

适用情况：

- 需求涉及异常处理
- 需求涉及任务、流程、状态恢复
- 需求涉及重试、失败、取消、恢复
- 需求涉及长时间运行或稳定性

关注点：

- 异常后是否可恢复
- 失败后状态是否一致
- 重复操作是否幂等
- 网络异常或服务异常如何处理

### 3. 易用性

适用情况：

- 需求涉及用户交互
- 需求涉及页面、操作流程、提示信息
- 需求影响用户理解和操作效率

关注点：

- 操作入口是否清晰
- 提示是否明确
- 错误反馈是否可理解
- 用户是否容易完成目标

### 4. 效率

适用情况：

- 需求涉及响应时间
- 需求涉及批量数据
- 需求涉及并发访问
- 需求涉及资源消耗
- 需求可能影响性能

关注点：

- 响应时间是否合理
- 批量场景是否可接受
- 并发下是否稳定
- 是否增加明显资源消耗

### 5. 维护性

适用情况：

- 需求涉及配置、规则、日志、可观测性
- 需求后续可能频繁调整
- 需求涉及复杂模块或多系统协作

关注点：

- 配置是否清晰
- 日志是否便于定位
- 错误信息是否可追踪
- 规则是否易于扩展

### 6. 可移植性

适用情况：

- 需求涉及多端、多浏览器、多设备、多系统环境
- 需求涉及部署环境变化
- 需求涉及兼容性或适配

关注点：

- 不同环境是否可用
- 不同设备表现是否一致
- 是否存在兼容性风险
- 配置迁移是否可靠

## 十一、测试点生成规则

测试点应该同时来源于：

```text
功能树
需求评审发现的问题
被选择的软件质量特性
历史缺陷或类似风险
数据、接口、状态、权限和依赖关系
变更影响范围和回归范围
```

测试点不要求写成完整测试用例。

但每个测试点应尽量包含：

```text
测试对象
测试关注点
测试思路
预期结果方向
关联质量特性
风险等级
测试优先级
是否建议回归
是否适合自动化
```

示例：

```text
测试对象：密码登录
质量特性：功能性
测试关注点：账号密码正确时是否可以登录成功
测试思路：使用有效账号和正确密码登录
预期结果方向：登录成功，进入首页，生成登录态
风险等级：高
测试优先级：P0
是否建议回归：是
是否适合自动化：是
```

测试优先级建议按以下规则判断：

```text
P0：核心流程、核心数据、权限安全、关键状态流转、上线阻塞风险
P1：重要功能、异常流程、边界场景、常见用户路径、历史缺陷相关场景
P2：低频功能、展示细节、辅助流程、体验优化类场景
```

是否建议回归，应根据以下因素判断：

- 是否影响旧功能
- 是否影响核心链路
- 是否关联历史缺陷
- 是否影响公共模块或公共接口
- 是否影响自动化脚本已有覆盖范围

是否适合自动化，应根据以下因素判断：

- 场景是否稳定
- 预期结果是否明确
- 测试数据是否可构造
- 执行频率是否较高
- 是否适合作为冒烟或回归场景

如果需求还不稳定，不建议过早标记为自动化候选。

## 十二、风险识别规则

报告需要输出风险提示。

风险来源包括：

- 需求缺失
- 需求歧义
- 状态不闭环
- 权限不明确
- 异常流程缺失
- 数据一致性风险
- 旧功能影响风险
- 回归范围遗漏风险
- 测试范围不清导致漏测或过测风险
- 外部系统依赖风险
- 接口和数据模型变更风险
- 自动化用例失效风险
- 性能或稳定性风险
- 可观测性不足

风险提示应该尽量具体，不要只写“存在风险”。

不建议输出：

```text
该需求存在一定风险。
```

建议输出：

```text
需求未说明任务取消后是否回退已变更的数据状态，可能导致取消后页面状态与后台数据不一致。
```

### 风险等级判定标准

为了避免风险等级随意标注，建议按以下规则判断。

高风险：

- 影响核心业务流程
- 影响数据一致性
- 影响权限、安全或合规
- 影响线上稳定性
- 影响资金、订单、任务、状态流转等关键对象
- 缺失信息会阻塞测试设计或上线判断
- 变更影响范围不清，可能导致核心旧功能漏回归

中风险：

- 影响局部功能
- 影响异常流程或边界场景
- 影响用户体验
- 影响部分角色或部分场景
- 缺失信息会影响测试覆盖范围
- 自动化用例可能需要调整，但不影响核心发布判断

低风险：

- 主要影响展示、文案、提示、低频辅助功能
- 不影响核心流程和关键数据
- 缺失信息只影响具体用例细节

风险等级需要结合业务影响和发生概率判断。

如果无法判断，应标记为“待确认”，不要强行给出确定等级。

## 十三、多文档冲突处理

真实项目中，不同文档之间可能存在冲突。

常见冲突包括：

- PRD 与 UI 原型不一致
- PRD 与开发设计不一致
- 接口文档字段与页面字段不一致
- 会议纪要与原始需求不一致
- 旧版本文档与新版本文档不一致
- 开发实现说明与验收标准不一致

Skill 不应自行选择某一份文档作为最终事实。

应按以下方式处理：

```text
识别冲突点
  ↓
列出冲突来源
  ↓
说明对测试分析的影响
  ↓
标记待确认优先级
  ↓
在测试点中避免使用未确认结论
```

输出格式建议：

| 优先级 | 冲突点 | 文档 A 描述 | 文档 B 描述 | 影响 | 建议确认对象 |
| --- | --- | --- | --- | --- | --- |
| P0/P1/P2 | 冲突内容 | 来源和描述 | 来源和描述 | 对测试设计的影响 | 产品/研发/设计 |

如果冲突影响核心流程、状态规则、权限规则或验收标准，应标记为 P0。

## 十四、证据来源和可追溯性

测试分析报告应尽量具备可追溯性。

也就是说，测试点不是凭空生成的，而应该能追溯到需求、设计、接口、原型或历史问题。

建议建立以下关系：

```text
输入文档
  ↓
需求点 / 规则点
  ↓
功能树节点
  ↓
质量特性
  ↓
测试点
  ↓
风险或待确认问题
```

测试点输出中建议增加“来源”字段。

示例：

| 功能模块 | 质量特性 | 测试关注点 | 来源 | 风险等级 |
| --- | --- | --- | --- | --- |
| 登录状态保持 | 可靠性 | Token 过期后是否重新登录 | PRD 登录章节 / 接口鉴权说明 | 中 |

如果输入文档没有明确来源编号，可以使用自然描述：

```text
来源：需求描述中的“验证码有效期 5 分钟”
来源：UI 原型登录页错误提示
来源：接口文档 login API 响应字段
来源：历史缺陷中登录状态丢失问题
```

如果某个测试点是基于经验推导，应标记为：

```text
来源：测试经验推导 / 类似历史风险
```

这样可以避免 AI 输出看起来很完整，但无法解释从哪里来的问题。

## 十五、信息完整度和分析置信度

报告中建议增加信息完整度和分析置信度。

信息完整度用于说明输入资料是否充分。

分析置信度用于说明当前分析结论是否可靠。

输出格式：

```text
信息完整度：高 / 中 / 低
分析置信度：高 / 中 / 低
原因：
- 已提供哪些关键资料
- 缺失哪些关键资料
- 哪些结论受缺失资料影响
```

建议判断标准：

高：

- 提供了需求文档、验收标准、关键设计或接口说明
- 主流程、异常流程、状态规则基本清楚
- 能够进行完整测试点分析

中：

- 提供了核心需求，但缺少部分设计、接口、原型或数据说明
- 可以分析主要功能和风险
- 部分测试点需要基于假设或待确认信息

低：

- 只有简单需求描述
- 缺少核心规则、验收标准或关键流程
- 不适合生成完整测试点

当置信度为低时，Skill 应优先输出待确认问题，而不是输出大量测试点。

## 十六、不同需求类型的分析策略

不同类型需求的测试分析重点不同。

Skill 应先识别需求类型，再调整分析重点。

### 1. 新功能需求

重点关注：

- 业务目标
- 主流程
- 异常流程
- 功能边界
- 验收标准
- 数据和状态闭环

### 2. 改造类需求

重点关注：

- 新旧逻辑差异
- 影响范围
- 历史功能回归
- 兼容旧数据
- 灰度和回滚

### 3. 接口类需求

重点关注：

- 参数规则
- 鉴权和权限
- 错误码
- 幂等
- 超时和重试
- 回调和异步通知
- 数据一致性

### 4. UI 交互类需求

重点关注：

- 页面入口
- 操作流程
- 字段规则
- 错误提示
- 空状态、加载状态、禁用状态
- 多端适配
- 易用性

### 5. 数据统计类需求

重点关注：

- 数据来源
- 统计口径
- 时间范围
- 去重规则
- 权限过滤
- 数据延迟
- 展示和导出一致性

### 6. 权限类需求

重点关注：

- 角色模型
- 可操作范围
- 越权访问
- 数据权限
- 接口权限
- 前后端权限一致性

### 7. 配置类需求

重点关注：

- 默认值
- 配置生效范围
- 配置变更后是否实时生效
- 配置冲突
- 配置回滚
- 操作日志

### 8. 异步任务类需求

重点关注：

- 任务创建
- 任务执行
- 任务取消
- 失败重试
- 超时处理
- 状态同步
- 补偿机制
- 日志和告警

### 9. 移动端需求

重点关注：

- 不同设备适配
- 网络切换
- 弱网
- 权限授权
- 前后台切换
- 安装升级
- 系统版本兼容

Skill 可以允许一个需求属于多个类型。

比如“移动端登录接口改造”同时属于移动端需求、接口类需求和改造类需求。

## 十七、输出模式

为了适配不同场景，建议支持三种输出模式。

### 1. 简版

适用场景：

- 快速需求评审
- 需求初稿分析
- 会议前准备

输出内容：

- 需求摘要
- 主要待确认信息
- 简要功能树
- 重点测试点
- 是否具备用例设计条件

### 2. 标准版

适用场景：

- 常规需求测试分析
- 用例设计前准备
- 迭代需求评审

输出内容：

- 完整需求摘要
- 可测试性判断
- 待确认信息
- 功能树
- 质量特性选择
- 测试点分析
- 风险提示
- 后续建议

### 3. 深度版

适用场景：

- 高风险需求
- 复杂系统需求
- 多系统协作需求
- 涉及权限、安全、数据一致性、状态流转的需求

输出内容：

- 标准版全部内容
- 多文档冲突分析
- 追溯关系
- 风险分级依据
- 需求类型专项分析
- 信息完整度和置信度
- 回归影响建议
- 专项测试建议

默认建议使用标准版。

如果用户未指定输出模式，Skill 可以根据需求复杂度自动选择，但需要在报告中说明选择原因。

## 十八、人工确认机制

当关键信息缺失或存在严重冲突时，Skill 不应继续强行生成完整测试点。

以下情况建议触发人工确认：

- 缺少核心需求描述
- 主流程不完整
- 核心业务规则缺失
- 验收标准不可判定
- PRD、UI、开发设计之间存在 P0 冲突
- 涉及权限但角色规则缺失
- 涉及状态流转但状态定义缺失
- 涉及接口但没有参数、错误码或鉴权说明
- 涉及数据统计但统计口径缺失
- 涉及高风险上线但没有回滚或灰度说明

触发人工确认时，输出应优先包含：

```text
无法继续完整分析的原因
必须确认的问题
缺失信息对测试设计的影响
建议确认对象
可以先进行的有限分析范围
```

这样既不浪费已有信息，也不会制造虚假的完整结论。

## 十九、可测试性判断

报告中建议加入可测试性判断。

输出格式：

```text
当前需求是否具备用例设计条件：是 / 否 / 部分具备
判断依据：
- 已明确的信息
- 缺失的信息
- 对测试设计的影响
```

判断规则：

- 如果主流程、核心规则、验收标准明确，可以判断为“是”
- 如果关键业务规则缺失，应判断为“否”
- 如果主流程明确但异常、边界、权限等信息缺失，应判断为“部分具备”

## 二十一、输出风格要求

Skill 输出应符合以下要求：

- 结论清晰
- 先指出不确定性，再生成测试点
- 不强行脑补需求
- 不输出与需求无关的大而全内容
- 不机械套用所有质量特性
- 测试点要能追溯到功能树或质量特性
- 待确认问题要说明影响
- 风险提示要具体
- 需要明确测试范围，不只列测试点
- 需要说明回归影响，不只分析新功能
- 需要标注测试点优先级
- 需要区分手工验证、回归建议和自动化候选
- 报告应适合测试、产品、研发共同阅读

## 二十二、约束规则

为了避免 AI 输出误导性内容，需要增加约束规则。

### 1. 不确定信息不得当作事实

如果需求未说明，应该写入待确认信息，而不是自行补全。

### 2. 不生成完整测试用例

该 Skill 输出测试点，不输出完整测试用例步骤。

完整用例设计可以作为下游 Skill。

### 3. 不默认六大特性全部适用

必须选择需要分析的质量特性，并说明原因。

### 4. 不忽略需求问题

如果需求缺失严重，应优先提示需求不可测试，而不是强行生成大量测试点。

### 5. 不输出内部敏感信息

如果输入中包含内部系统名、地址、账号、接口密钥、客户信息等，不应在公开报告中扩散。

### 6. 不忽略文档冲突

如果不同输入文档存在冲突，必须显式输出冲突点和待确认问题。

### 7. 不隐藏低置信度

如果资料不足，应说明信息完整度和分析置信度，不应输出看似确定的结论。

## 二十三、Skill 输出质量验收标准

后续如果把该设计落成正式 Skill，可以用以下标准评审输出质量。

| 验收项 | 合格标准 |
| --- | --- |
| 需求摘要 | 能准确概括业务目标、对象、核心流程和预期结果 |
| 需求评审 | 能识别缺失、歧义、不闭环和不可判定内容 |
| 待确认信息 | 问题具体，并说明影响和建议确认对象 |
| 功能树 | 结构清晰，能反映主要功能、子功能和关键状态 |
| 质量特性选择 | 只选择必要特性，并说明选择或不选择原因 |
| 测试点分析 | 测试点能追溯到功能树、质量特性或输入文档 |
| 测试范围 | 能说明重点测试、轻量验证、暂不覆盖和回归范围 |
| 回归分析 | 能识别旧功能、历史缺陷、核心链路和自动化资产影响 |
| 优先级判断 | 能区分 P0/P1/P2，并给出合理依据 |
| 自动化候选 | 能判断哪些测试点适合后续自动化 |
| 风险提示 | 风险具体，有等级和判定依据 |
| 文档冲突处理 | 能显式列出冲突来源和影响 |
| 可测试性判断 | 能判断是否具备用例设计条件，并说明依据 |
| 置信度说明 | 能说明信息完整度和分析结论可信程度 |
| 约束遵守 | 不脑补、不机械套用、不输出敏感信息 |

不合格输出示例：

```text
需求整体清晰，建议覆盖功能测试、性能测试、安全测试、兼容性测试。
```

原因：

- 没有指出依据
- 没有识别待确认信息
- 机械套用测试类型
- 没有功能树和风险分析

合格输出应该具体说明为什么测、测什么、不测什么、还缺什么。

## 二十八、设计结论

这个 Skill 的设计是合理的。

它把 AI 介入点放在需求阶段，而不是等到需求完全确定后才生成用例。

它的核心价值是：

- 提前发现需求问题
- 降低测试设计误差
- 让测试点有结构来源
- 避免 AI 机械生成大而全内容
- 让需求评审、功能树和质量特性形成闭环

后续如果创建正式 Skill，建议先实现轻量版本，再根据真实需求分析结果逐步补充示例、模板和脚本。
