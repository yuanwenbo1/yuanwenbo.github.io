# 测试资产沉淀与知识库治理 Skill 设计文档

## 一、设计背景

测试过程中会产生大量零散信息。

这些信息包括：

- 测试账号
- 测试数据
- 需求分析报告
- 测试策略报告
- 测试用例
- 用例评审记录
- 接口集合
- 自动化脚本
- 测试环境说明
- 配置说明
- 缺陷记录
- 回归清单
- 检查清单
- 项目经验
- 会议纪要

这些内容如果没有结构化沉淀，通常会散落在聊天记录、临时文档、表格、测试平台、代码仓库、接口工具和个人笔记中。

结果是：

- 下次类似需求又要重新分析
- 测试账号和测试数据找不到
- 历史缺陷经验没有进入回归
- 用例和自动化脚本无法对应
- 环境配置靠人记忆维护
- 测试报告写完就失效
- 新人无法快速理解项目测试经验
- AI 后续分析缺少可靠上下文

因此，需要一个 Skill 帮助测试人员在测试过程中或测试结束后，把零散知识整理成固定资产。

这个 Skill 的价值不是“写总结”，而是把一次性测试产物转化为长期可复用的测试资产和知识库内容。

## 二、Skill 定位

建议名称：

```text
测试资产沉淀与知识库治理 Skill
```

建议英文名：

```text
test-asset-consolidation
```

核心定位：

```text
识别、整理、分类和归档测试过程中产生的账号、数据、文档、用例、缺陷、接口集合、脚本、环境配置、检查清单和项目经验，并沉淀为可复用、可维护、可追溯、可治理的测试资产。
```

该 Skill 不只是版本结束后的总结能力。

它也可以在测试过程中持续使用，用于维护测试资产目录。

## 三、适用场景

适合使用该 Skill 的场景包括：

- 项目测试结束后的资产整理
- 新需求测试过程中沉淀账号、数据和文档
- 历史缺陷经验转回归资产
- 测试用例库整理
- 接口测试集合归档
- 自动化脚本和手工用例关联
- 测试环境和配置说明整理
- 测试知识库建设
- 新人交接资料整理
- AI 测试上下文资产治理

不建议直接用于：

- 保存明文密码、Token、密钥
- 直接公开内部系统地址和客户信息
- 替代安全的凭证管理系统
- 无筛选地归档所有临时资料
- 把一次性产物全部当成长期资产

## 八、敏感级别设计

资产需要标记敏感级别。

### 公开

可以放到公开博客、公开知识库或对外分享。

例如：

- 通用测试方法
- 通用检查清单
- 脱敏后的经验总结

### 内部

只能团队内部使用。

例如：

- 内部项目测试策略
- 内部模块风险清单
- 内部测试流程说明

### 受限

只允许指定角色查看。

例如：

- 测试账号说明
- 环境配置说明
- 数据库结构说明

### 敏感

不能出现在普通文档中，只能记录索引或凭证存放位置。

例如：

- 密码
- Token
- 密钥
- 客户数据
- 内部系统敏感地址

## 十、沉淀流程

建议 Skill 按以下流程执行。

```text
读取测试过程资料
  ↓
识别可沉淀资产
  ↓
按资产类型分类
  ↓
判断复用价值和敏感级别
  ↓
生成资产登记表
  ↓
给出知识库归档建议
  ↓
提出维护计划
  ↓
输出测试资产沉淀报告
```

## 十二、约束规则

- 不记录明文密码、Token、密钥。
- 不公开内部系统地址、客户信息和敏感数据。
- 不把一次性资料全部沉淀为长期资产。
- 不把未验证信息写成团队标准。
- 资产必须有来源、用途和维护建议。
- 敏感资产只记录索引或安全存放位置。

## 十三、输出质量验收标准

| 验收项 | 合格标准 |
| --- | --- |
| 分类完整 | 能覆盖账号、数据、文档、用例、接口、缺陷、环境和清单 |
| 资产可用 | 每项资产有用途、来源和存放位置 |
| 敏感治理 | 能识别敏感资产并避免明文泄露 |
| 可维护 | 有维护人、维护频率或后续动作 |
| 可追溯 | 能说明资产来源于哪个项目、模块或测试过程 |
| 可复用 | 能判断资产复用价值和适用场景 |
| 可反哺 | 能说明资产如何反哺测试分析、策略、用例和评审 |

## 十四、与其他 Skill 的关系

```text
需求测试分析 Skill
  ↓
测试策略设计 Skill
  ↓
测试用例设计 Skill
  ↓
测试用例评审 Skill
  ↓
测试资产沉淀与知识库治理 Skill
  ↓
反哺前序 Skill
```

该 Skill 会反哺：

- 需求测试分析：提供历史风险、模块经验和检查清单
- 测试策略设计：提供回归范围、风险画像和历史缺陷
- 测试用例设计：提供历史用例、测试数据和回归场景
- 测试用例评审：提供覆盖标准、缺陷经验和评审清单
