# 测试用例评审 Skill 设计文档

## 一、设计背景

AI 可以快速生成大量测试用例，但用例数量不等于测试质量。

低质量用例常见问题包括：

- 前置条件不清楚
- 测试步骤不可执行
- 测试数据不可构造
- 预期结果不可判定
- 多条用例重复覆盖同一场景
- 只覆盖主流程，遗漏异常和边界
- 优先级与风险不匹配
- 与需求、测试点、历史缺陷无法追溯

如果这些用例直接进入执行阶段，会导致执行效率低、结果不可信、回归维护困难。

因此，需要一个测试用例评审 Skill，在用例进入执行前进行质量检查。

## 二、Skill 定位

建议名称：

```text
测试用例评审 Skill
```

建议英文名：

```text
test-case-review
```

核心定位：

```text
对测试用例进行质量评审，识别不可执行、不可判定、重复冗余、覆盖遗漏、优先级不合理和追溯关系缺失的问题，并输出评审结论和修改建议。
```

该 Skill 不负责重新生成完整测试用例。

它的重点是评审和改进建议。

## 三、适用场景

适合使用该 Skill 的场景包括：

- 测试用例评审会前检查
- AI 生成用例后的质量审查
- 用例入库前检查
- 回归用例集治理
- 自动化候选用例筛选前检查
- 历史用例清理和去重
- 用例覆盖追溯检查
- 测试资产质量评估

不建议直接用于：

- 需求完整性评审
- 测试策略制定
- 完整用例生成
- 自动化脚本生成
- 测试执行结果分析

## 四、输入信息

### 1. 核心输入

```text
测试用例集
```

测试用例集应尽量包含：

- 用例 ID
- 所属模块
- 用例标题
- 前置条件
- 测试数据
- 操作步骤
- 预期结果
- 优先级
- 用例类型
- 来源
- 是否回归
- 是否冒烟
- 是否自动化候选

### 2. 推荐输入

- 需求测试分析报告
- 测试策略设计报告
- 测试用例设计报告
- 功能树
- 测试点清单
- 质量特性选择
- 业务规则和验收标准
- 历史缺陷
- 历史回归用例
- 自动化用例集

### 3. 输入缺失处理

如果只有用例集，没有需求、测试点和策略输入，Skill 只能评审可执行性、可判定性和格式质量。

如果缺少需求测试分析报告，则无法完整判断覆盖遗漏。

如果缺少测试策略报告，则无法准确判断优先级和测试深度是否合理。

如果缺少历史缺陷，则无法判断历史问题回归覆盖是否完整。

这些限制需要在评审报告中明确说明。

## 五、输出目标

Skill 输出一份结构化测试用例评审报告。

报告应包含：

- 评审结论
- 输入信息完整度
- 用例整体质量评价
- 问题用例清单
- 覆盖缺口
- 重复和冗余用例
- 不可执行用例
- 不可判定用例
- 优先级不合理用例
- 自动化候选风险
- 修改建议
- 是否建议进入执行阶段

## 六、评审流程

建议 Skill 按以下流程执行。

```text
读取测试用例集
  ↓
读取需求测试分析和测试策略
  ↓
检查输入信息完整度
  ↓
检查用例字段完整性
  ↓
检查可执行性和可判定性
  ↓
检查覆盖完整性
  ↓
检查重复、冗余和可合并用例
  ↓
检查优先级和用例类型
  ↓
检查来源追溯关系
  ↓
检查自动化候选合理性
  ↓
输出评审报告和修改建议
```

## 七、评审维度

### 1. 字段完整性

检查每条用例是否具备必要字段：

- 用例标题
- 所属模块
- 前置条件
- 测试数据
- 操作步骤
- 预期结果
- 优先级
- 用例类型
- 来源

缺少关键字段的用例，应标记为不完整用例。

### 2. 可执行性

检查点：

- 前置条件是否清楚
- 操作步骤是否完整
- 步骤顺序是否合理
- 测试数据是否可构造
- 测试环境或依赖是否明确
- 执行人是否可以按步骤复现

不可执行用例示例：

```text
验证用户登录功能正常。
```

问题：

- 没有前置条件
- 没有测试数据
- 没有操作步骤
- 没有明确预期

### 3. 可判定性

检查点：

- 预期结果是否明确
- 是否能判断通过或失败
- 是否避免模糊表述
- 是否说明页面、接口、数据、状态或日志结果

不建议的预期结果：

```text
功能正常
结果正确
符合预期
页面显示无误
```

建议改成：

```text
登录成功后跳转到首页，接口返回 code=200，页面右上角显示当前用户名。
```

### 4. 覆盖完整性

检查是否覆盖：

- 主流程
- 异常流程
- 边界条件
- 权限场景
- 状态流转
- 数据一致性
- 历史缺陷
- 回归范围
- 质量特性要求
- P0 风险点

如果需求测试分析报告中存在 P0 测试点，但用例集中没有对应用例，必须标记为高风险覆盖缺口。

### 5. 优先级合理性

检查 P0 / P1 / P2 是否合理。

P0 应覆盖：

- 核心业务流程
- 核心数据写入
- 权限安全边界
- 关键状态流转
- 冒烟用例
- 上线准入强相关场景

如果低风险展示类用例标为 P0，或核心链路用例标为 P2，都应指出。

### 6. 重复和冗余

检查点：

- 多条用例是否只换标题但步骤相同
- 是否重复覆盖同一个输入组合
- 是否可以合并为参数化用例
- 是否存在低价值重复验证

重复用例不一定直接删除，也可以建议合并或参数化。

### 7. 用例类型合理性

检查用例类型是否标注正确。

常见类型：

- 冒烟
- 主流程
- 异常
- 边界
- 权限
- 状态流转
- 数据一致性
- 回归
- 兼容
- 易用性
- 自动化候选

类型标注错误会影响后续执行计划和回归范围。

### 8. 追溯关系

检查用例是否能追溯到：

- 需求点
- 功能树节点
- 测试点
- 质量特性
- 风险点
- 历史缺陷
- 测试策略

如果用例无法追溯，应标记为“来源不清”。

来源不清的用例不一定无效，但需要确认其存在价值。

### 9. 自动化候选合理性

检查自动化候选是否合理。

适合自动化：

- 高频回归
- 核心冒烟
- 预期明确
- 数据可构造
- 执行路径稳定
- 接口层可验证

不适合立即自动化：

- 需求频繁变化
- UI 不稳定
- 需要强人工判断
- 测试数据难稳定
- 低频且维护成本高

如果自动化候选标注不合理，应输出调整建议。

## 八、问题分级

评审问题建议按 P0 / P1 / P2 分级。

### P0：必须修复

包括：

- 核心 P0 风险无用例覆盖
- 用例不可执行
- 预期结果不可判定
- 关键业务规则遗漏
- 权限或安全场景遗漏
- 状态流转核心路径遗漏

### P1：建议修复

包括：

- 主要异常或边界遗漏
- 优先级不合理
- 测试数据不够明确
- 用例类型标注错误
- 来源追溯不完整

### P2：可优化

包括：

- 标题不够规范
- 用例描述可读性一般
- 低价值重复用例
- 可合并或参数化优化

## 九、评审结论规则

评审结论建议分为三类。

### 通过

条件：

- P0 覆盖完整
- 无不可执行用例
- 无不可判定用例
- 主要风险已覆盖

### 部分通过

条件：

- P0 基本覆盖
- 存在 P1/P2 问题
- 不影响进入执行阶段，但需要补充或优化

### 不通过

条件：

- P0 风险缺口明显
- 多条核心用例不可执行
- 预期结果普遍不可判定
- 用例无法支撑测试策略

## 十、输出模式

### 简版

适合快速检查。

输出：

- 评审结论
- 主要问题
- P0 覆盖缺口
- 修改建议

### 标准版

适合常规评审。

输出：

- 输入完整度
- 整体质量评价
- 问题用例清单
- 覆盖缺口
- 重复冗余
- 优先级问题
- 修改建议

### 深度版

适合复杂需求或用例资产治理。

输出：

- 标准版全部内容
- 需求追溯矩阵
- 测试点覆盖矩阵
- 历史缺陷覆盖
- 自动化候选评审
- 用例库治理建议

- 评审结论：通过 / 部分通过 / 不通过

- 问题用例数：

## 十二、约束规则

- 不只给“通过/不通过”结论，必须说明依据。
- 不直接重写全部用例，优先输出问题和修改建议。
- 不把需求缺失造成的问题归咎为用例问题。
- 对 P0 覆盖缺口必须明确指出。
- 缺少上游资料时，需要说明评审范围受限。
- 涉及内部敏感信息时必须脱敏。

## 十三、输出质量验收标准

| 验收项 | 合格标准 |
| --- | --- |
| 结论清晰 | 能给出通过、部分通过或不通过结论 |
| 问题具体 | 能定位到具体用例和具体问题 |
| 覆盖评审 | 能识别需求、测试点、风险和历史缺陷覆盖缺口 |
| 可执行性评审 | 能识别前置条件、步骤、数据不清问题 |
| 可判定性评审 | 能识别预期模糊和不可判断问题 |
| 优先级评审 | 能指出优先级与风险不匹配问题 |
| 追溯评审 | 能说明用例来源是否清楚 |
| 修改建议 | 建议具体可操作 |

## 十四、与其他 Skill 的关系

```text
测试用例设计 Skill
  ↓
测试用例评审 Skill
  ↓
测试数据设计 Skill
  ↓
测试执行辅助 Skill
```

测试用例评审 Skill 的输出，会影响是否进入测试执行，也会反向修正用例设计和测试数据设计。
