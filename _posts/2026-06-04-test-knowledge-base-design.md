# 测试知识库从 0 搭建与治理设计文档

## 一、设计背景

在测试工作中，真正有长期价值的内容往往不是某一次测试报告本身，而是测试过程中沉淀下来的业务理解、测试经验、历史缺陷、测试账号、测试数据、接口资料、环境配置、用例资产、自动化脚本和专项检查清单。

但这些内容通常分散在不同地方：

- 需求文档和设计文档
- 测试用例平台
- 接口调试工具
- 自动化工程
- 缺陷管理系统
- 群聊记录
- 个人笔记
- 临时表格
- 项目复盘文档
- 本地文件夹

如果没有一个统一的知识库承载这些信息，测试资产就很容易变成一次性材料。
项目结束后，经验没有复用；新人加入后，学习成本很高；需求再次出现时，仍然要重新分析；AI 想辅助测试时，也缺少可靠的上下文。

因此，需要设计一个从 0 搭建的测试知识库体系，用来承载测试资产，并形成长期可维护、可检索、可复用、可治理的质量知识基础。

## 二、知识库定位

测试知识库不是简单的文档仓库，也不是把所有资料随意堆进去。

它应该承担四个角色：

1. 测试资产仓库
2. 项目质量经验库
3. 团队测试规范库
4. AI 测试上下文库

知识库的核心目标不是“存得多”，而是“找得到、看得懂、用得上、能维护、可追溯”。

它应该服务于以下对象：

- 测试工程师
- 测试开发工程师
- 产品经理
- 研发工程师
- 项目负责人
- 新加入团队成员
- AI Agent 或测试分析 Skill

## 三、建设目标

测试知识库需要实现以下目标：

- 统一承载测试过程中产生的关键资产
- 将零散资料整理成稳定的知识结构
- 支持按项目、模块、流程、资产类型检索
- 支持历史经验反哺后续需求分析和测试设计
- 支持测试账号、测试数据、接口资料、环境配置的安全索引
- 支持历史缺陷和风险经验进入回归体系
- 支持团队测试规范、模板和检查清单持续演进
- 支持 AI 读取高质量上下文，提高分析和生成质量
- 支持知识的更新、归档、废弃和权限治理

## 四、从 0 搭建步骤

### 1. 明确知识库边界

第一步不是建目录，而是先明确知识库要承载什么、不承载什么。

建议纳入知识库的内容：

- 业务规则
- 模块说明
- 需求分析报告
- 测试策略报告
- 测试用例资产
- 用例评审记录
- 测试数据说明
- 测试账号索引
- 接口说明和接口集合索引
- 自动化脚本说明
- 环境配置说明
- 历史缺陷经验
- 回归检查清单
- 专项测试方法
- 项目复盘
- 测试规范和模板
- AI 可读取的上下文资料

不建议直接纳入知识库的内容：

- 明文密码
- Token
- 私钥
- 证书
- 客户隐私数据
- 未脱敏日志
- 内部敏感系统地址
- 没有来源的口头结论
- 无复用价值的一次性材料

### 2. 选择知识库载体

知识库可以使用不同载体，不同阶段可以逐步演进。

早期低成本方案：

- GitHub Pages
- Git 仓库 + Markdown
- Notion
- 飞书知识库
- 语雀
- Confluence

更适合测试工程团队的方案：

- Git 仓库管理 Markdown 文档
- 文档站点用于阅读
- CI 用于检查链接、格式和敏感词
- 搜索功能用于快速检索
- 权限系统用于控制敏感内容
- AI 检索层用于给 Agent 提供上下文

如果目标是让 AI 深度参与测试工作，建议优先选择结构化程度高、版本可追踪、便于检索和导出的载体。

## 六、知识库初始化流程

从 0 搭建时，不建议一开始追求大而全。
更合理的方式是先建立最小可用知识库。

### 第一阶段：最小可用

目标：先让团队有地方放、能找到、能复用。

需要完成：

- 建立基础目录
- 编写知识库使用说明
- 编写文档命名规范
- 编写资产登记模板
- 建立测试账号索引模板
- 建立测试数据模板
- 建立项目测试资产模板
- 建立历史缺陷经验模板
- 建立回归检查清单模板

### 第二阶段：项目接入

目标：把一个真实项目的测试过程接入知识库。

需要完成：

- 为项目创建独立目录
- 接入需求分析报告
- 接入测试策略报告
- 接入测试用例资产
- 接入测试数据说明
- 接入测试账号索引
- 接入接口资料
- 接入缺陷经验
- 接入环境说明
- 接入项目复盘

### 第三阶段：流程固化

目标：让知识库成为测试流程的一部分。

需要完成：

- 需求分析完成后沉淀需求分析资产
- 测试策略完成后沉淀范围和风险判断
- 用例设计完成后沉淀用例资产
- 用例评审完成后沉淀评审问题
- 测试执行完成后沉淀测试数据和环境问题
- 缺陷关闭后沉淀缺陷经验
- 项目结束后沉淀复盘和回归规则

### 第四阶段：AI 接入

目标：让知识库能够被 AI 用来提升测试分析质量。

需要完成：

- 为核心文档增加摘要
- 为资产增加标签
- 为文档增加来源和更新时间
- 建立 AI 可读取的上下文索引
- 标记哪些内容可以给 AI 使用
- 标记哪些内容禁止外发或必须脱敏
- 建立 Prompt 和 Skill 使用规则

## 七、知识库治理规则

知识库建起来之后，真正的难点是治理。
如果没有治理，知识库会很快变成新的资料垃圾堆。

### 1. 准入规则

进入知识库的内容需要满足至少一个条件：

- 后续项目可复用
- 能降低新人理解成本
- 能帮助定位历史问题
- 能提升测试设计质量
- 能支持回归测试
- 能作为 AI 分析上下文
- 能形成团队规范

不满足这些条件的临时信息，不应该长期沉淀。

### 2. 命名规则

建议统一命名格式：

```text
项目名-模块名-资产类型-主题.md
```

示例：

```text
配送系统-任务调度-历史缺陷-任务状态异常.md
配送系统-账号权限-测试账号索引.md
配送系统-接口测试-任务创建接口说明.md
```

### 3. 标签规则

建议至少包含以下标签维度：

- 项目
- 模块
- 资产类型
- 测试阶段
- 质量特性
- 风险等级
- 敏感级别
- 是否可被 AI 使用

### 4. 权限规则

知识库内容需要按敏感级别管理：

```text
公开：可以对外公开的通用方法论和经验。
内部：团队内部可见的项目经验和测试资料。
受限：仅项目成员或授权人员可见的账号、环境、接口信息。
敏感：不能直接进入普通知识库，只记录安全存放位置。
```

### 5. 更新规则

每一类资产都需要有维护周期：

- 业务规则：需求变更后更新
- 测试账号：每月或权限变更后检查
- 测试数据：每次大版本后检查
- 接口资料：接口变更后更新
- 自动化脚本：脚本变更后更新
- 缺陷经验：缺陷关闭后更新
- 回归清单：版本发布前更新
- 项目复盘：项目结束后更新

### 6. 废弃规则

过期知识比没有知识更危险。

以下内容需要标记废弃：

- 已下线模块
- 已失效接口
- 已失效账号
- 已废弃测试数据
- 已不适用的环境配置
- 与当前业务规则冲突的历史说明
- 无法追溯来源的结论

废弃内容不要直接删除，建议移动到归档区，并记录废弃原因。

## 八、AI 使用规则

测试知识库未来应该服务于 AI，但不能无边界地把所有内容都交给 AI。

### 1. 可供 AI 使用的内容

- 需求摘要
- 功能树
- 测试策略
- 测试点
- 用例结构
- 历史缺陷模式
- 回归检查清单
- 测试方法论
- 已脱敏的业务规则
- 已脱敏的项目经验

### 2. 不应直接给 AI 的内容

- 明文账号密码
- Token
- 密钥
- 证书
- 客户隐私数据
- 未脱敏日志
- 内部敏感地址
- 涉及商业机密的原始材料

### 3. AI 上下文索引

建议为 AI 单独维护上下文索引：

```text
文档名称：
适用任务：
可用范围：
敏感级别：
推荐读取章节：
禁止读取章节：
摘要：
最后更新时间：
```

这样 AI Agent 在执行需求分析、测试策略设计、用例生成和用例评审时，可以更准确地知道应该读取什么，不应该读取什么。

## 九、与测试流程 Skill 的关系

测试知识库是底座，测试流程 Skill 是能力。

可以理解为：

```text
测试知识库提供上下文
        ↓
需求测试分析 Skill 输出需求分析资产
        ↓
测试策略设计 Skill 输出策略资产
        ↓
测试用例设计 Skill 输出用例资产
        ↓
测试用例评审 Skill 输出评审资产
        ↓
测试资产沉淀 Skill 整理并回写知识库
        ↓
知识库反哺下一轮测试分析
```

其中，测试资产沉淀 Skill 是知识库和测试流程之间的连接器。
它负责把一次测试活动中产生的零散产物转化为稳定资产，再写回知识库。

## 十、知识库治理角色

建议定义以下角色：

```text
知识库 Owner：
负责整体结构、规范、治理策略和质量检查。

项目维护人：
负责具体项目下的测试资产维护。

资产维护人：
负责账号、数据、接口、脚本、环境等具体资产的有效性。

评审人：
负责检查重要知识是否准确、可复用、可公开。

AI 使用审核人：
负责判断哪些内容可以作为 AI 上下文，哪些必须脱敏或禁止使用。
```

小团队可以一人多角色，但角色职责要清楚。

## 十一、质量检查机制

知识库需要定期检查质量。

建议检查以下问题：

- 是否存在无标题、无来源、无维护人的文档
- 是否存在过期账号或过期数据
- 是否存在无法复用的一次性记录
- 是否存在重复文档
- 是否存在敏感信息泄露风险
- 是否存在与现行规则冲突的旧知识
- 是否存在没有进入回归体系的高价值缺陷经验
- 是否存在 AI 可用但未建立索引的上下文

可以每月做一次轻量检查，每个大版本后做一次重点检查。

## 十二、落地建议

从实际落地角度，建议不要一次性建设完整系统，而是按以下顺序推进：

1. 先建立 Markdown 知识库目录。
2. 先沉淀一个真实项目。
3. 先把测试账号、测试数据、历史缺陷、回归清单管起来。
4. 再把需求分析、测试策略、测试用例、用例评审结果接入。
5. 再建立 AI 上下文索引。
6. 最后再做搜索、权限、自动检查和知识库看板。

如果一开始就追求平台化，很容易变成重工具、轻内容。
测试知识库的第一价值是让团队的测试经验能够留下来，并在下一次测试中真正被使用。

## 十三、设计结论

测试知识库是测试资产沉淀的承载基础。

测试资产沉淀 Skill 负责把测试过程中产生的零散信息整理成资产，测试知识库负责保存、组织、检索、复用和治理这些资产。

当知识库建设成熟后，它不仅能提升团队协作和新人上手效率，也能成为 AI 参与测试工作的上下文基础。

因此，后续测试体系可以围绕以下闭环演进：

```text
测试过程产生资产
测试资产沉淀 Skill 整理资产
测试知识库承载资产
AI 和测试人员复用资产
新项目继续产生更高质量资产
```

这个闭环一旦建立，测试能力就不再只依赖个人经验，而会逐渐变成团队可复用、可积累、可进化的质量资产。
